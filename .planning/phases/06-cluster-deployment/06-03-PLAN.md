**Step 2: Prepare smoke test file**

Create a debian:buster container tar for the smoke test. This is a deliberately old image with hundreds of known vulnerabilities — ideal for verifying the full scan pipeline:
```bash
docker pull debian:buster
docker save debian:buster -o /tmp/smoke-test-debian-buster.tar
gzip /tmp/smoke-test-debian-buster.tar
```
The resulting file (~50MB) will be uploaded via the API.

**Step 3: Get or create an API key**

Check if an API key is available. The smoke test needs `Authorization: Bearer <key>` with sufficient roles for upload + job creation. Options:
- Check cluster secrets: `ssh dg-node1 "kubectl get secret scanrook-secrets -n scanrook -o jsonpath='{.data}'" | base64 -d`
- Or generate one via the scanrook.io UI (Settings > API Keys > Create)
- The key needs OPERATOR role or higher (JOB_WRITE_ROLES includes OPERATOR bit 4)

If no API key is readily available, the smoke test can be run manually via the UI as a fallback (the checkpoint task covers this).

**Step 4: Run API-driven smoke test**

Using curl and jq, execute the full pipeline:

```bash
API_BASE="https://scanrook.io"
API_KEY="<retrieved-key>"
TEST_FILE="/tmp/smoke-test-debian-buster.tar.gz"

# 1. Get presigned upload URL
PRESIGN=$(curl -s -X POST "$API_BASE/api/uploads/presign" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"filename":"smoke-test-debian-buster.tar.gz","contentType":"application/gzip"}')
echo "Presign response: $PRESIGN"

# Extract fields
UPLOAD_URL=$(echo $PRESIGN | jq -r '.url')
OBJECT_KEY=$(echo $PRESIGN | jq -r '.key // .fields.key')
BUCKET=$(echo $PRESIGN | jq -r '.bucket // "uploads"')

# 2. Upload test file to S3 via presigned URL
# The presign response includes fields for multipart form upload
# Adapt based on actual presign response structure
curl -X POST "$UPLOAD_URL" \
  -F "key=$(echo $PRESIGN | jq -r '.fields.key')" \
  -F "bucket=$(echo $PRESIGN | jq -r '.fields.bucket // empty')" \
  -F "Content-Type=application/gzip" \
  -F "file=@$TEST_FILE"

# 3. Create scan job
JOB=$(curl -s -X POST "$API_BASE/api/jobs" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d "{\"bucket\":\"$BUCKET\",\"object_key\":\"$OBJECT_KEY\",\"filename\":\"smoke-test-debian-buster.tar.gz\"}")
JOB_ID=$(echo $JOB | jq -r '.id')
echo "Job created: $JOB_ID"

# 4. Poll job status until done or failed (max 5 minutes)
for i in $(seq 1 60); do
  STATUS=$(curl -s "$API_BASE/api/jobs/$JOB_ID" \
    -H "Authorization: Bearer $API_KEY" | jq -r '.status')
  echo "Poll $i: status=$STATUS"
  if [ "$STATUS" = "done" ] || [ "$STATUS" = "failed" ]; then
    break
  fi
  sleep 5
done

# 5. Verify findings exist (should be 300-500 for debian:buster)
if [ "$STATUS" = "done" ]; then
  FINDINGS_COUNT=$(curl -s "$API_BASE/api/jobs/$JOB_ID/findings?limit=1" \
    -H "Authorization: Bearer $API_KEY" | jq '.total // .items | length')
  echo "Findings count: $FINDINGS_COUNT"
fi
```

If the API key approach fails (auth issues, quota limits), note the failure and defer to the manual checkpoint below.

If the smoke test fails at any step:
- Check worker logs: `ssh dg-node1 "kubectl logs -n scanrook -l app=scanrook-worker --since=10m"`
- Check job status in DB: the job may be stuck in "running" or marked "failed"
- If the deployment itself is broken, use rollback: `ssh dg-node1 "kubectl rollout undo deployment/scanrook-worker -n scanrook"`
